#### Why This Matters
Previously, the logic for calculating "Arrive By" windows could be inconsistent between the recurrence definition and the generated trips. This change ensures that the time windows are calculated once during setup and then reliably applied to all trips in the series, improving scheduling accuracy.

### üìÖ Store and Apply Pickup/Dropoff Windows for Recurring Trips

<Warning>This feature is currently categorized as 'Do Not Enable - In Development' and is not yet ready for customer use. Please await communications from the Product team on a feature rollout plan.</Warning>

**What's New**  
When you create or edit a recurring trip, the system now calculates and permanently stores the precise pickup and dropoff windows associated with it. This ensures that all individual trips generated from the recurrence use these pre-calculated windows, improving timing consistency, especially for Arrive By trips.

**Pre-requisites**
- Requires `arriveByV2ForRecurringTrips` feature flag.
- Requires Admin access.

<Accordion title="See More">

#### Where You'll See It
- This is a backend change with effects visible in *Requests > Request Recurrences*. While the new data fields are not directly displayed in the UI, their impact is seen when individual trips are created from a recurrence.

#### What You Can Do
- Create or update a recurring trip, and the system automatically calculates and saves the associated pickup/dropoff windows.
- If you modify the recurrence (e.g., change the time or location), the windows are automatically recalculated and saved.
- When an individual request is generated from this recurrence, it will now use the stored windows rather than calculating new ones.

#### Why This Matters
Previously, the time windows for each instance of a recurring trip were calculated upon creation, which could lead to inconsistencies if service settings changed over time. Storing the windows with the recurrence itself ensures all generated trips adhere to the exact timing constraints defined when the recurrence was first configured.
</Accordion>
